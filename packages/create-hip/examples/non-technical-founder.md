# HIP.md — Human Interface Protocol

## Human Operating Mode

Role in this repo: Non-technical founder
Technical fluency: Lower technical depth; strong product, domain, and business knowledge
Primary intent: Ship a feature
Autonomy level: Make changes, pause before structural decisions
Explanation depth: Step by step
Risk tolerance: Balanced
Decision style: Tradeoffs then recommendation
Feedback style: Gentle

## Technical Fluency

| Domain | Level |
|--------|-------|
| Code | Low |
| Git | Low |
| Terminal | Low |
| Debugging | Low |
| Deployment | Low |
| Architecture | Medium |
| Database | Low |
| Dependencies | Low |

## Approval Required Before

- authentication
- database schema
- environment variables
- deployment configuration
- payment flows
- legal or policy text
- security rules
- migrations
- public-facing copy

## Review Workflow

- show a plain-language summary of what changed and why
- list any risks or things that could break
- provide manual steps I can follow to verify the feature works
- ask for approval before moving to the next phase

## Definition of Done

Work is not done until:

- the feature works and I can test it myself
- you have explained what changed in plain language
- risks are stated clearly
- next steps are clear
- I can explain the change to someone else

## Update Rule

If the human corrects the same collaboration behavior twice, ask whether HIP.md should be updated.
